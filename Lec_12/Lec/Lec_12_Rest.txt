* Representational State Transfer (REST)

    REST е една от най-важните концепции в съвременното програмиране на уеб услуги

    REST е стил софтуерна архитектура за реализация на уеб услуги, а не е конкретна технология, която се инсталирва на даден компютър
    # Това е като набор от правила и добри практики това как компютърите да си говорят помежду си през интернет 
    REST не е спецификация или протокол, HTTP е протокол, докато REST е начина, по който използваме този протокол

    Състои се от шест принципа, които допринасят към това уеб услугата да бъде проста, бързодействаща и скалируема
    # Основната цел на REST е да се подобри и улесни комуникацията между две системи

    Уеб услугите, които покриват REST принципите, се наричат RESTful уеб услуги.

* Принципи на REST архитектурата

    има шест REST принципа:
    - REST е клиент-сървър архитектура
    # Клиентът примерно ще се грижи как изглежда изображението, докато server-а разполага с даннтие и логиката
    - REST предоставя единен интерфейс
    # Това означава, че начинът на комуникация е стандартизиран, без значение какво е устройството той изпозлва същите команди като GET, POST и други 
    - REST не пази състояние (stateless)
    # Сървърът не помни какво си правил в предишната стъпка и цялата заявка, която клиента праща трябва да съдържа цялата информация, за да бъде разбран и изпълнен
    - REST предлага възможности за кеширане (cacheable)
    # Информацията, която сървърът връща, може да бъде маркира като "кешируема", това позволява да се запази отговора и при следващото отговаряне да бъде заредена мигновенно от паметта
    - REST може да бъде многослойна система
    # Има много междинни слоеве в комуникаята с клиента и сървъра, това позволява повече сигурност и разпределение на натоварването
    - REST може да предостави code on demand
    # Възможно е и сървърът да върне върне освен данни, но и изпълним JavaScript код, който клиента да стартира

* REST е клиент-сървър архитектура 

    Клиентът и сървърът са отделни компоненти при REST архитектурата
    # Идеята е че клиентът и сървърат са отделни компоненти с напълно различни задчи

    Separation of concerns - клиентът и сървърът имат различни задачи
    # Сървърът съдържа бизнес логиката както и съхранението на данните, докато клиентът извлича даннит от сървъра и се грижи за представянето им
    # Позволява извършването на CRUD операции (Create, Read, Update, Delete) върху ресурсите

    Единният интерфейс позволява разделянето на клиент и сървър.
    # Това позволява да се променят клиентът и сървъра независимо един от 
    # Единствено не трябва да се променя начина, по който те си комуникират

* Ресурси 

    REST въвежда концепцията за ресурс, това означава, че всяка информация, която може да бъде именувана, може да бъде ресурс

    Реално програмистите определят кои са ресурсите, с които ще се оперира уеб услугата
    # Един ресурс може да бъде html страница, Js файлове или изображение

* REST предоставя единен интефейс 

    REST предоставя единен интерфей между клиента и сървъра за комуникация помежду си

    Има четири конкретни условия, които оформят този единен интефейс:
    - Идентификация на ресурсите
    # Един ресурс може да се идентифицира уникално използвайки UR
    - Промяна на ресурси посредством техните представяния
    # Клиентът не може директно да промени базата данни на сървъра, клиента получава представяне на този ресус (примерно JSON или XML) и ако той иска да го промени праща промененото състояние към сървъра и той си решава
    - Описателни представяния на ресурси и съобщения
    # Всяка съобщение (дали заявка или отговор) трябва да съдържа достатъчно информация само по себе си, за да може получателят да го разбере и обработи, това се случава чрез headers и други
    - Hypermedia as the engine of application state
    # Идеята е че когато сървърт връща отговор не само да включва данните но и хипервръзки към други свързнаи действищ или ресурси

* Идентификация на ресурси

    В web програмирането ресурс е всякакъв обект информаицп (примерно файл, потребител, стаии) и всеки такъв ресурс има свой уникален адрес наречене URI (Uniform Resource Identifier)

    Използвайки различни HTTP методи (GET, PUT, DELETE и др.) се извършват въпросните операции върху ресурса.

    Примери за достъпване на ресурс: $ curl -X GET https://api.github.com/repos/octocat/Hello-World
    # curl е web browser но без графичен интерфейс, като идва от Client URL, позволява на клиент да говори със сървър чрез интернет адрес (URL)
    # curl само тегли суовият код, който би се показал на браузъра, но само го показва като текст в конзолата 
    # curl е най-добрия начин да се тества дали едно API работи,

    Други примери:
    # Get a repo
    - $ curl -X GET https://api.github.com/repos/octocat/Hello-World

    # Update a repo
    - $ curl -X PATCH https://api.github.com/repos/octocat/Hello-World -d "$REQUEST_BODY"

    # Delete a repo
    - $ curl -X DELETE https://api.github.com/repos/octocat/Hello-World

* Представяне на ресурси

    Ресурсите са концептуално разделени от представянията, които се връщат към клиента
    # Това означава, че ресурсът (същността на данните) е едно, а начинът, по който го вижда клиента е друго 

    Сървъра пази ресурсите си в база от данни и на файловата система, а връща на клиентите ресурси под формата на HTML, JSON и XML
    # Сървъра може да си пази данните в сложни SQL таблици или файлове, но когато ги праща на клиента ги праща в четим формат като JSОН

    Не винаги върнатото представяне е вътрешното представяне
    # Реално данните са едни и същи просто е показано по различен начин на клиента както и на сървъра
    
    Ресурсите трябва да имат еднакви представяния в отговорите от сървъра
    Ползвателите на REST API-то трябва да използват тези представяния, за да модифицират състоянието на ресурса в сървъра
    # за да се променят ресурси в сървъра, трябва да се изпратят във сищия формат (примерно json) обратно към сървъра, а не директно пишеш в базата данни 

* REST != HTTP 

    В практиката често термините HTTP и REST са взаимнозаменяеми
    # Модерният REST API-та се изграждат върху HTTP протокола и затова се нарича REST API 

    HTTP е най-използваният протокол за имплементиране на RESTful уеб услуги, но REST не налага използването на HTTP
    
    Немалко HTTP API-та използват бисквитки (cookies), за да пазят някакво състояние. Това нарушава принципа, че REST не пази състояние
